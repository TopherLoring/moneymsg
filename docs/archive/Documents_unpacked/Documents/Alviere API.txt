Authentication
Alviere uses the openid authentication standard.

This section details what is required for authentication and access to the Alviere endpoints.

Get API Credentials
In order to connect to the API, you’ll need to have authentication credentials that should be provided by your Relationship Manager.

If you don’t have authentication credentials yet, please request them by sending an email from your company email to support@alviere.com.

Api Authentication
In order to authenticate to the platform you'll need the following information:

client_id: Your unique client ID in the Alviere platform
client_secret: Your API key used to connect to the APIs; you may have more than one secret for your account
username: The username that will be used for authentication
password: The password that will be used for authentication
Authentication Flow
The following details the flow for authentication:

To call any of the APIs in the platform you'll need an access token, so this is always the service you need to call first
The response object from the authentication endpoint will return multiple fields as per the JWT specification. Below are the most relevant:
access_token: The token that is used for all subsequent calls to the API
expires_in: The time limit after which the access token will expire; in seconds
On all subsequent calls to the API, you'll need to include a header in the HTTP request with that information.

The format of the header is Authorization: Bearer [you_access_token]

Accounts
This section outlines the integration with Alviere to create and manage your customers and users, and to perform KYC and Identity Verification on them.

A user account is a representation of a customer of your program, which is required for the following Alviere products:

FBO Accounts - Alviere provides FDIC-insured bank accounts through its banking partners so you can provide full banking services to your customers. The funds are held for benefit of your customers and Alviere manages all day-to-day activities of the banking program, so you can focus on further extending your business offering.
Stored Value Accounts - Alviere provides e-wallet / stored value accounts through its banking partners. This means any user account can store money in an e-wallet and easily transfer it from e-wallet to e-wallet, or to/from payment methods.
Direct Deposit Accounts - With Alviere's FDIC-insured FBOs and/or stored value account services, you can provide your customers with a DDA routing number where they can deposit their salaries or receive ACH payments from external sources, giving you another touchpoint for promotional and revenue opportunities.
User accounts are created and validated in the Alviere platform in stages. Each stage provides different permissions to what actions the user can perform within the Alviere platform.

KYC and Identity Verification
KYC and Identity Verification within the Alviere platform is a multi-stage process and is, by nature, the validations that need to be performed as an asynchronous process. During this process the user account will transition between several statuses. Please check the User Status section for more details.

When submitting a request for a new user account, you must provide all of the relevant information that is defined for your program, including name, date of birth, address, email address, phone number, and Social Security Number, depending on the products you are enabling through the Alviere platform. This information must be present on the payload of your user creation API request.

User Status
The user status field indicates the current status of your users in the Alviere platform.

Some statuses are internal and represent the current stage in the identity verification process, while other statuses represent the need to obtain additional information from the customer or that a manual review of the identity is required.

The following statuses are possible for users in the Alviere platform:

Status	Description
CREATED	Prior to KYC and Identity Verification checks, this is the status of a user account once you submit a successful user creation request.
ACTIVE	If the user is fully active in the system, their status will be ACTIVE. In some cases, the active status does not mean that the user will have full access to all the products provided by Alviere.
INACTIVE	This status indicates that all permissions to access the system are currently revoked and the user will not be able to perform any actions in the Alviere platform until you choose to reactivate the user account.
PROCESSING	This status indicates that identity verification is in progress; no action is required while the user account is in this status.
PENDING_USER	This status indicates that further action is required from the user. This status could occur due to missing or incorrect data on the user's profile such as an address or Social Security Number. For more information, see the Pending User Status Reasons section below.
MANUAL_REVIEW	This status indicates that the user account is currently under manual review by an operator. The stage property will indicate why the user account is under manual review (e.g. if stage: "FRAUD" this means that the account_abuse score was too high and needs to be validated by the operator)
REJECTED	This status indicates the user account has been rejected by the platform, either manually or automatically. The reason for the rejection will be returned on the reason property of the user.
DELETED	This status indicates the user account has been deleted and removed from the platform. The user account data will still be present in the system, but the accout cannot be reinstated. If you wish to temporarily remove access to a user account, you should change the status to INACTIVE instead
Processing Stages
The status of pending verification applies to users that have not yet been completely identified.

The possible status reason for the pending verification stage property are:

Stage	Description
FRAUD	This stage indicates that the fraud check internal process is in progress.
ADDRESS	This stage indicates that the address provided by the user is being validated.
IDENTITY	This stage indicates that the identity verification process is being performed. Information such as the date of birth, address, and/or Social Security Number are being matched against public and platform-private records for accuracy.
DOCUMENT	This stage indicates that the photo identification, provided by the user to validate their identity, is being processed.
SANCTIONS	This stage indicates that the internal process of Sanctions and Negative News screening is not complete.
(empty)	This stage indicates that the user has not started the KYC validation or is on-boarded and there's no more stages to execute.
User Status Reasons
The status_reason field provides a descriptive reason for the current status of the user.

If the status is PENDING_USER, it means more information is required from the user in order to proceed with the Identity Verification process. The following status reasons can occur for this status:

Status Reason	Description
INVALID_NAME	The first and/or last name provided are not valid and one or both need to be re-submitted.
INVALID_DOB	The provided date of birth is not valid; this could happen for programs where there is a minimum age requirement or the date of birth does not match the public record information on the user.
INVALID_ADDRESS	The user has provided an address that is invalid and needs to be re-submitted.
INVALID_PHONE	The user has provided a phone that is invalid and needs to be re-submitted.
INVALID_LAST_4_SSN	The user has provided the last 4 digits of their Social Security Number that could not be matched against public and platform-private records. This will generally result in the user account being placed in status_reason: 'REQUIRES_FULL_SSN'. Note: This status is only valid for US consumers.
INVALID_SSN	The user has provided their Social Security Number that could not be matched against public and platform-private records. This will generally result in the user account being placed in status_reason: 'REQUIRES_FULL_SSN'. Note: This status is only valid for US consumers.
REQUIRES_LAST_4_SSN	The user is required to provide the last 4 digits of their Social Security Number in order to proceed with the identity verification process. This status usually happens when it was not possible to lookup the last 4 digits of their SSN from public records. Note: This status is only valid for US consumers.
REQUIRES_FULL_SSN	The user is required to provide their full 9-digit Social Security Number in order to proceed with the identity verification process. This status usually happens when it was not possible to lookup the full SSN from public records. Note: This status is only valid for US consumers.
INVALID_DOSSIER	The user has provided a dossier that is invalid and needs to be re-submitted.
EXPIRED_DOCUMENT	The user has provided a document that is expired. A valid document needs to be submitted to proceed with the identity verification process.
If the status is REJECTED, the user account either failed some automated checks or was manually rejected by an Operator from the Alviere platform. The following status reasons can occur for this status:

Status Reason	Description
NO_IDENTITY_MATCH	The user has failed to provide information to validate their identity.
SANCTIONED_USER	The user has been positively identified with OFAC Sanctions or Negative News.
FRAUD	The user has been marked fraudulent; for example, a Chargeback was filed for this account and the operator has marked it as fraud.
CUSTOM	The user was rejected and the operator has provided a custom description for the rejection.
Dossiers
The dossier structure of the User Account is an essential part of Identity Management. In most programs enabled by Alviere, documental identification might be required to enable the full set of services for end users. When a document is required in order to proceed, the Pending User Status will be REQUIRES_PHOTO_ID.

Dossiers need to be validated through external partners of Alviere; this process is also asynchronous. The following dossier statuses are applied:

Dossier Status	Description
CREATED	This is the status of a dossier once it has been submitted through the API indicating that the dossier is created.
PENDING	This status indicates that the dossier is under verification.
MANUAL_REVIEW	This status indicates that the dossier is under manual verification.
VERIFIED	This status indicates that the dossier has been approved by the system and the user has been positively identified.
REJECTED	This status indicates that the dossier has been rejected. Possible causes range from false documents to bad quality of photos.
FAILED	This status indicates that the dossier has been failed. This happens when the documents format is invalid.
EXPIRED	This status indicates that the dossier submitted has expired and the user account has been placed back in PENDING_USER status in order for the user to submit documentation with a valid expiration date.
DELETED	This status indicates that the dossier is deleted.
Add Account
Creates a user account in the system.

This service will automatically trigger the KYC and Identity Management process.

Please note that a user account cannot be duplicated within the same business account; a user is considered duplicated when the same external_id is provided.

header Parameters

required
string
Example: Bearer eyJhbGciOiJSUzI1NiIsInR5cCIgOiAiSldUI...
The access token.

Request Body schema: application/json
required
external_id
required
string
The ID of this user in your own platform. This will be associated with this account and can be used to identify the account as well as protect our API against duplicate accounts.

NOTE: This field MUST be unique. Any subsequent Create Users with same external_id will fail with a HTTP 409.

personal_information
required
object
This structure contains the personal information of a user account.

primary_address	
object
The Primary address for this user account.

Notes:

If you do not pass this field, you have to call the Add Address aftewards to add the primary address for this account

Account Wallet
This section details the ledger which contains the funding information for the account.

Ledger and Balances
There are three types of funding buckets managed by the Alviere ledger:

Bucket	Description
BALANCE	These are the settled funds in the user account.
IN_TRANSIT	The unsettled funds in the user account. This amount generally includes funds that have been authorized by a user transaction, but have not yet settled from the corresponding processor.
CAPTIVE	The funds that have been held captive for a transaction that was performed on the account. For example, if an ACH Debit was performed, the funds will be held captive until the transaction executes.
The account wallet APIs allow you to perform operations on the account ledger or move funds within the system to other account wallets.

Account Wallet Transactions
Transactions on the account wallet follow a specific flow depending on the transaction type being executed.

The available account statuses are as follows:

Status	Description
ACTIVE	The transaction has been created in the system.
PROCESSING	The transaction is currently processing; this usually happens when a ledger transaction requires a payment to be pulled or pushed from a payment method.
PENDING	The transaction is pending acceptance from the receiver of the transaction.
CANCELLED_USER	The transaction has been cancelled at the user's request.
CANCELLED_SYSTEM	The transaction has been cancelled by an automated system process; the reason for the cancellation will be present on the response of the API call.
ERROR	The transaction has failed due to a system error.
FAILED	The transaction failed due to an unsuccessful underlying payment transaction, such as an NSF response from the processor. The reason for the failure will be present on the response of the API call.
EXPIRED	The transaction has expired in the system because the processing of the transaction did not complete within the predefined time for execution.
COMPLETED	The transaction has been completed and the funds were successfully transferred.
REJECTED	The transaction was rejected by a manual review process.
PENDING_KYC	The transaction is currently pending and awaiting the receiver/recipient to complete their KYC information.
PROCESSING_PAYMENT	The transaction is currently processing a payment operation with the underlying processor.
WAITING	The transaction is currently waiting for funds to be available in order to be executed; this status occurs only on Withdraw transactions when the corresponding reserve account has insufficient funds. The transaction will automatically post when the funds on the reserve account are replenished.
MANUAL_REVIEW	The transaction has been placed in manual review and awaits an operator decision to be posted.
VOIDED	The transaction has been voided due to voiding of the underlying payment transaction directly with the processor.
Note: Different workflows and statuses may apply to different transaction types.

Holding Funds
In order for any user account to hold funds in Alviere, they need to perform the full KYC and Identity Management processes. For further details, see the Accounts section of the documentation.

Service Fees
Alviere supports the application of Service Fees on any user account wallet transaction. To apply your own service fees to a transaction, you should set the desired amount or percentage of the total value of the transaction in the corresponding structure.

Service Fees will be automatically be moved to your service fees account; this is a native feature of Alviere's Treasury Management module. A transaction of type SERVICE_FEE will be automatically created as a child transaction of the original user account wallet transaction. Service Fees are only collected once the transaction moves to COMPLETED status. If the transaction is not completed (eg. REFUND), the service fee transaction will also be void and the funds will not move to your service fees account.

Service Fees in Alviere always apply to transactions and can be charged in the following types:

Service Fee Type	Description
PERCENT	Use this service fee type if you want a service fee percentage to be applied on the total amount of the transaction. For example, if the total amount is $100 and you want to apply a 10% fee, you should set the property to value: 10
AMOUNT	Use this service fee type if you want a fixed amount service fee to be applied to the transaction, in cents (eg. value: 1000).
PERCENT_CAPPED	Use this service fee type if you want a service fee percentage to be applied on the total amount of the transaction, up to a certain amount. For example, if the total amount is $1000 and you want to apply a 10% fee capped to $50, you should set the value: 10 and cap: 5000
In terms of how the transaction amount applies to the transaction:

Charge Type	Description
UPCHARGE	The service fee will be applied as an upcharge to the transaction. For example, if a customer is performing a cross-border transaction of $100 and there's a PERCENT service fee of 10%, the user account would be charged $110.
DEDUCT	The service fee will be applied as a deduction to the transaction amount. For example, if a customer is performing a withdraw transaction of $100 and there's a PERCENT service fee of 10%, the user would only receive $90 on their account.
Captivating/Uncaptivating Funds
Alviere supports the custom captivation of funds in the user account's wallet. Captivation needs to have a corresponding ledger entry associated with it, generally a LOAD funds transaction type.

The following business rules apply to funds captivation:

Funds can only be captivated in the user account wallet if the sum balance + in_transit funds is equal to or greater than the funds being captivated.
When captivating funds, Alviere's ledger will always use balance funds first, and if the full amount is not available, the remainder will be taken from the in_transit funds.
The following business rules apply to the uncaptivation of funds:

Funds can only be uncaptivated when the amount is less than or equal to the captive amount and the original transaction that captivated the funds has not completed.
Funds that are uncaptivated return to the corresponding buckets:
To the balance bucket for settled funds;
To the in_transit bucket for unsettled funds.
Note that the captivated funds also abide by the settlement rules of the underlying payment processor. For example, if funds are captivated from the in_transit bucket and settle in the meantime, they return to the user account wallet as balance when the funds are uncaptivated.

Load Funds

Alviere supports loading of funds into user account wallets using the Load Funds service.

In order to load funds into the account wallet, the user account needs to have a valid payment method, tokenized in the system through the Add Card or Add Bank Account services. Note that loading funds from a Bank Account requires that a Plaid token is present and valid in the system.

Once funds are loaded in the user account, the account wallet will have the corresponding amount in the in_transit bucket. This means that the loading of funds transaction has been accepted, but the funds are not settled yet. Funds will be settled according to the selected payment method settlement times.

Withdraw Funds
Alviere supports withdrawing funds from the user account's wallet to an external funding destination using the Withdraw Funds service.

In order to withdraw funds from the account wallet, the user account needs to have a valid payment method, tokenized in the system through the Add Card or Add Bank Account services. Note that withdrawing funds to a Card requires that the push-to-card service is active and the card supports this feature over the debit card rails of the network schemes.

Once funds are withdrawn from the user account, the account wallet will have the corresponding amount in the captive bucket. This means that the withdrawing of funds transaction has been accepted, but the funds are not available in the destination source yet. Funds will be settled according to the selected payment method settlement times.

Important Note: Although withdrawing of funds returns structure of the underlying payment transaction, account wallet operations can only be cancelled using the unique transaction_uuid returned by the Withdraw Funds service.

Send Funds
Alviere provides the ability to send funds to another user account. This feature is only available for customers subscribed to a P2P program. Funds can only be sent to user accounts that are in ACTIVE status.

The funds will be sent using the following business logic:

If the user has enough balance to send the funds, the funds from this bucket will be used first.
If the user doesn't have enough balance to send the funds, and the sum of balance + in_transit funds in the user account wallet are enough to perform the transaction, Alviere will use the full balance funds and the remaining funds will be used from the in_transit bucket.
If the user does not have the full amount of funds to send in either or both buckets, Alviere will use the provided payment method to perform a transaction of the full amount of the funds.
If the user does not have enough funds to perform the transaction in either or both bucket, and no payment method is provided, the transaction will fail.
Note that for the sending of funds using a payment method, the transaction could fail due to any payment processing error, such as NSFs or transactions not authorized.

Tokenization
This section outlines the endpoints related to our tokenization service.

Card Tokenization Process
When a card is added to our system through our tokenization service, we perform multiple tests before encrypting and storing the card details with the highest PCI standards.

The following flow needs to be respected at all times:

Add a card to the system using the create token API. All card information needs to be submitted to add a card.
Check the CVV results section. Invalid CVVs will result in unusable cards.
Make sure the AVS results are acceptable for your business scenario, and if not, notify your user that the card could not be added.
When successful, all information gathered and generated to validate the card is returned together with the newly created token.
Make sure you store this token securely. This is the only way to reference this card in the future and to perform all card-related operations.
Cards could have the following statuses:

Status	Description
CREATED	The Card was created in the system and is currently being validated.
ACTIVE	The Card is active and can be used for transactions.
REJECTED	The Card has been rejected.
FAILED	This is the status when a Card is failed in the system.
EXPIRED	This is the status when a Card is expired in the system.
DELETED	This is the status when a Card is deleted in the system.
Card Validations
In order to complete the validation, the following checks are performed for all cards:

BIN Check: We retrieve card details such as issuing country, card brand (Visa, Mastercard, Discover, etc...) and card type (credit, debit, prepaid, etc...). Although this information is retrieved based on a reliable BIN database, this information is subject to constant changes by the network, regulators and other parties, so if you have any limitations as per the types of cards your service should accept, make sure you validate this information regularly.
AVS Check: This Address Verification System (AVS) check is used to verify the card data, such as billing address and ZIP code, against the supported card network’s billing information of the cardholder. This validation may have various results, depending on the provided information. Please see the AVS Match Responses documentation below for more details.
CVV Check: This check verifies that the security code (CVV) submitted by the customer matches the security code associated with the card. See documentation below for the CVV Check Response Codes.
$0 Authorization: This standard network card verification process ensures that the card is fully operational. It does not show up in the account holder’s statement and does not need any void or refund operation. If you are using Alviere’s tokenization services, this transaction is done automatically when you add a card to the system, and will perform CVV and AVS checks for you.
Card Failures
If a card fails any of the validation checks above, the response will remain “201 CREATED” and the response payload will contain the information regarding the results and the main reason for the failure. You may access this information at any time. Please note that the cards remain encrypted and stored in our system as per compliance regulations.

If a card validation check fails, and you wish to retry (eg. the customer has missed the CVV), you can issue a new create card request with the correct parameters, and a new tokenization of this card will be returned to you with an updated validation status.

Cancel Transaction
The Cancel endpoint is either a Void or a Refund for a given card transaction.

A transaction can be voided only if it’s within the timeframe allowed by the card networks; if the transaction could not be voided, a new transaction will need to be created as a refund.

header Parameters
Authorization
required
string
Example: Bearer eyJhbGciOiJSUzI1NiIsInR5cCIgOiAiSldUI...

Request Body schema: application/json
required

required
string <uuid>
This field is Alviere's internal unique transaction identifier and is required to reference the transaciton that we wish to perform an operation on, such as VOID or REFUND

operation_type
required
string (Cancel operation type)
Enum: "VOID" "REFUND"
The Cancel operation type can be one of the following:

VOID: This operation type will attempt to perform a void on the given transaction. A transaction can be voided only if it’s within the timeframe allowed by the card networks and will always require a network request. If a void is no longer possible, you can use a refund operation type instead.

REFUND: This operation type will try to perform a refund on a transaction that can no longer be voided. If a transaction is in a state that can be voided, the refund will not be possible.

Bank Processing
This section describes how to execute payment ACH transactions, such as ACH Debits and ACH Credits.

ACH is processed in Alviere based on the NACHA processing windows. Alviere posts transactions Monday to Friday at 12:15pm, except for official US bank holidays.

There are two types of transactions supported for ACH:

Same-Day ACH: Transactions processed before 12pm Eastern will be received/debited on the same day as the transaction.
Next-Day ACH: Transactions processed after 12pm Eastern will be received/debited the day after the date of the transaction.

Card Issuance
This section describes how to issue and manage cards with the Alviere platform.

To issue cards, you'll need to have a Cardholder Program with Alviere.

Card Types
The Alviere platform supports the following types of cards:

Type	Description
DEBIT	This is a type of card that uses the balance of the user account wallet in order to authorize transactions.
PREPAID	This is a type of card that needs to be preloaded with a balance in order to authorize transactions.
GIFT CARD	This is a type of card that has a fixed balance preset at the moment of the issuance of the card.
The main difference between a Debit or Prepaid card and a Gift Card is the latter does not require KYC to be performed.

Card Products
In the Alviere platform, each card on a program is identified by a unique product_id that will be provided to your company at the moment of the program setup.

A product_id is always associated to only one of the card types mentioned above. Your program can support any number of card configurations and Alviere will assign individual product_id for each one of the cards.

Product IDs also define if the card is issued virtually or physically, the design/card art of the card, carrier design (physical cards only), and all the settings, limitations, and fees of the card. These settings will be defined at the moment of the setup of your Cardholder Program, and cannot be set using the API.

Shipping Address Rules
The shipping address defines where a card is shipped to via postal service. This setting does not apply to virtual card products.

Virtual card product types will not have this field present.
This field is mandatory for Gift Card product types.
For non-Gift Card programs, if this property is present, the card will be shipped to this address instead of the address found on the user account
Card Statuses
The following table details the issued card statuses in the Alviere platform:

Type	Description
CREATED	The card was created in the Alviere platform; this is the initial status for all card issuance requests.
PENDING	The card is currently pending; this could happen due to a secondary ID verification.
PROCESSING	The card is currently processing.
ACTIVE	The card is active and ready to be used.
FROZEN	The card is currently frozen and all transaction authorizations will be declined.
SET_TO_EMBOSS	The card is set to be embossed by the manufacturing partner; this status is only applicable to physical card products.
SHIPPED	The card has been shipped via mail; this status is only applicable to physical card products.
READY_TO_ACTIVATE	The card is ready to be activated and for the PIN to be set; this status is only applicable to physical card products.
RETURNED_MAIL	The card was returned via the postal service; this status is only applicable to physical card products.
LOST_STOLEN	The card has been reported by the cardholder as stolen or lost; this is a final status.
CANCELED	The card has been canceled and can no longer be used; this is a final status.
EXPIRED	The card has expired; this is a final status.
Incentives
Alviere supports incentivization/bonuses for all of its cards.

Two types of incentives are supported:

Type	Description
CASHBACK	The cardholder will receive into the balance of their account, a cashback of the defined amount/percentage after a successful authorization of a card transaction.
BOOST	The card authorization will succeed if the balance + boost amount/percentage are equal or lower to the amount being authorized. For example, if the card balance is $100 and there's a BOOST incentive of 10%, the system will authorize a transaction up to $110.
Incentives can be configured individually for each card issued through the API. You can configure an amount of the incentive and/or the valid merchants where the incentive is to be applied.

The following rules are applied to the incentives:

Percentage: This property defines the percentage of the authorization amount to apply the incentive (BOOST or CASHBACK)
Cap: This property defines a cap for the incentive (BOOST or CASHBACK); for example, if you have a 10% cashback with a $5 cap, and the authorized transaction is $100, the value of the cashback into the cardholder's balance will be $5
You can also limit the merchants where the incentives are to be applied; i.e. "only apply this incentive on purchases for merchant X". Merchant incentives can be applied to specific merchant IDs or to merchant names.

In the case of merchant names, the system will do a match of the name; i.e. it will be applied as long as the merchant name exists in the merchant description field. For example, if this property is set to "Alviere", then a transaction for "Alviere, Inc." and a transaction for "Alviere Financial Services" will both apply.

Allowed Merchants
You can also specify that the card balance can only be spent on specific merchants; i.e. the card was issued with a balance that can only be spent on specific merchant(s).

In the case of merchant names, the system will do a match of the name; for example, if this property is set to "Alviere", then a transaction for "Alviere, Inc." and a transaction for "Alviere Financial Services" will both apply.

Reporting
Use the Reporting API to get all transactions processed by Alviere and to check the statuses of transactions, such as their settlement status.

Below are the statuses that will be returned from the API on the status property:

Status	Description
CREATED	We have received the transaction and accepted it for processing.
IN_TRANSIT	The transaction has been authorized by the network and has not yet settled.
FAILED	The transaction authorization has failed; the details of the reason for the failure will be in the details of the response.
VOIDED	This transaction was voided by the merchant before it settled.
SETTLED	The transaction has been settled and the funds are on their way to the merchant's banks.
COMPLETED	This is a status that happens after the settlement (ie. transactions in this status are already settled); in future versions of the API this will mean that the webhook was called.

Webhooks
Webhooks can be used to get events from Alviere. For example, when a transaction is settled, Alviere can send an event to your webhook consumer with information about the transaction.

Events are grouped into Subscriptions and each Subscription event will be posted into a URL provided by you at the moment of the subscription.

Note: Alviere sends all events asynchronously so that it doesn't block or slow down any system operation or transaction. All events are actions that already took place.

Subscriptions
Alviere provides three different subscription types available:

ACCOUNT provides information about the lifecycle of consumers’ accounts
TRANSACTION provides information about the lifecycle of transactions
ISSUED_CARD provides information about the lifecycle of all Cards Issued by Alviere
BANK provides information about the lifecycle of all Banks Issued by Alviere

Authentication
Each subscription requires the following information:

Subscription Type
Target endpoint (needs to be a complete HTTPS URL)
The required security key (you must setup one and give us)
When we call the desired HTTPS URL, we will include an HEADER called Alviere-Auth and the value of this HEADER will be the security key you provided to us.

Processing
Each time we send a message for the provided URL, we expect a 200 HTTP status code, otherwise we will retry the event applying a polynomial backoff starting at 20ms and going up to 2 minutes after each retry.

NOTE that we will continuously retry sending the message and this will impact any other messages in the queue as we strictly maintain messages in order (FIFO).

How to subscribe
