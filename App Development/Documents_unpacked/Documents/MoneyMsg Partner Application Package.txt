# 1. Executive Overview

MoneyMsg is an Android-native conversational payments platform.

The product operates as a **system keyboard using Android's InputMethodService API**, enabling users to initiate peer-to-peer payments directly within any messaging conversation.

Supported messaging environments include:

* SMS
* RCS
* WhatsApp
* Signal
* Telegram

The experience replicates the functionality of Apple Cash within iMessage for the Android ecosystem.

Users can:

* load funds into a wallet
* send money within a conversation
* receive payments via a claim link
* withdraw funds to a bank account or debit card

The architecture uses a **closed-loop wallet model** to minimize network costs and reduce fraud exposure.

# 3. Product Description

MoneyMsg solves a UX problem in mobile payments.

Most P2P systems require users to:

1. leave a conversation
2. open a payment app
3. complete a transaction
4. return to messaging

MoneyMsg eliminates that context switch.

Payments are initiated directly inside the conversation through the system keyboard.

Example interaction:

```
User A typing in chat
↓
opens MoneyMsg keyboard panel
↓
enters $20
↓
keyboard injects payment link
↓
recipient taps link
↓
payment claimed
```

# 5. Transaction Flows

## Wallet Load

### Card Load

User debit card
→ card processor
→ wallet credit

Purpose: fund wallet balance.

---

### Bank Load

User bank account
→ bank transfer network
→ wallet credit

Purpose: fund wallet balance.

---

## Peer-to-Peer Transfer

Sender wallet
→ locked transfer intent
→ payment link shared in chat
→ recipient claims payment
→ wallet-to-wallet transfer executed.

---

## Cash Out

Wallet funds
→ external debit card or bank account.

# 6. Compliance Model

The platform operates using a **Banking-as-a-Service architecture**.

Compliance responsibilities are distributed as follows:

| Function       | Provider |
| -------------- | -------- |
| KYC / AML      | Alviere  |
| OFAC screening | Alviere  |
| Wallet ledger  | Alviere  |
| Card rails     | TabaPay  |
| Bank rails     | Dwolla   |

The platform does not store sensitive identity data.

Sensitive data handled by providers:

* SSN
* card numbers
* bank credentials

# 7. Data Handling Policy

Sensitive financial data is never stored locally.

The platform stores only tokenized references.

Stored fields include:

| Data             | Description           |
| ---------------- | --------------------- |
| processor_token  | payment routing token |
| institution_name | bank name             |
| mask             | last four digits      |

Data protection measures:

* TLS encryption
* secure token storage
* restricted database access
* audit logging

# 8. Fraud & Risk Controls

The system implements multiple safeguards.

### Identity Verification

User onboarding requires KYC verification.

---

### Balance Locking

Funds are deducted when a transfer intent is created.

This prevents double spending.

---

### Intent Expiration

Payment links expire after 24 hours.

Unclaimed funds automatically return to the sender.

---

### Transaction Monitoring

The backend logs all financial operations including:

* loads
* transfers
* withdrawals

Suspicious activity can trigger account freezing.
