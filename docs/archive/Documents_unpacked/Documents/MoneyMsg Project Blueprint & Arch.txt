# MoneyMsg Project Blueprint & Architecture Guide (Revised)

## I. Executive Summary

MoneyMsg is a third-party Android keyboard and companion application designed to facilitate instant peer-to-peer (P2P) transactions natively within RCS messaging environments. Operating as a subsidiary project under laundr.me, the platform utilizes a closed-loop digital wallet architecture. By isolating internal money movement from external funding networks, MoneyMsg achieves zero-cost peer-to-peer transfers while generating revenue strictly through external on-ramp and off-ramp convenience fees.

## II. Project Goals & Problem/Solution

**The Problem:** Traditional P2P payments require users to exit their active messaging context to use a disparate financial application. Furthermore, routing instant payments directly between external bank accounts or debit cards incurs significant network fees and fraud risk, compressing platform margins.

**The Solution:** MoneyMsg acts as an OS-level `InputMethodService` (custom keyboard), allowing users to generate and inject transaction intents directly into their RCS chat thread without switching apps. To solve the margin compression, the platform employs a Wallet-Ledger Paradigm. Users are onboarded into a compliant Banking-as-a-Service (BaaS) ledger. Internal transfers between these wallets cost the platform nothing, allowing precise monetization on the funding and cash-out legs.

## III. Technology Stack

The stack prioritizes cross-platform flexibility, mobile-first workflows, and high-performance offline/server-side execution.

* **Backend Runtime & API:** Bun (high-throughput, zero-dependency environment).
* **Database:** PostgreSQL (utilizing strict `NUMERIC(15,4)` typing to prevent floating-point anomalies).
* **Companion Client:** React Native (Web/iOS/Android compatible for user onboarding, KYC, and deep link resolution).
* **Keyboard Client:** Native Android (Kotlin extending `InputMethodService` to meet strict OS memory/performance boundaries).

## IV. Infrastructure & Partner Providers

The vendor architecture assigns highly specific operational roles to minimize legal overhead and engineering complexity.

| Provider | Operational Role | Protocol / Mechanism |
| --- | --- | --- |
| **Alviere** | Ledger & Compliance | Money Transmitter License (MTL) coverage, KYC/AML, FBO account issuance, internal zero-cost book transfers. |
| **Plaid** | Authentication | External account linking and secure `processor_token` generation to avoid PCI scope. |
| **TabaPay** | Card Routing | Unified API for instant push/pull via Account Funding Transactions (AFT) and Original Credit Transactions (OCT). |
| **Dwolla** | Bank Routing | Instant Account-to-Account (A2A) push/pull via the RTP® Network and FedNow® Service. |

## V. Revenue Model & Economics

The platform monetizes external liquidity while absorbing zero cost for internal movement.

* **Platform Fee:** 1.75% + $0.50 applied to all external money movement.
* **Wallet Load (User Pays):** Gross charge calculated to yield the exact net wallet balance. Platform absorbs the TabaPay AFT or Dwolla RTP fee from this margin.
* **Cash Out (User Pays):** Fee deducted directly from the gross withdrawal amount. Platform pays the TabaPay OCT or Dwolla Push fee from this margin.
* **Internal P2P (Free):** Transacting settled funds within the Alviere ledger incurs $0.00 network cost and $0.00 user fee.

## VI. Transaction User Flows

Because the Android keyboard cannot deterministically know the recipient's identity at the moment of typing, all P2P sends utilize an asynchronous "Intent-and-Claim" mechanism via deep links injected into the RCS thread.

### Flow 1: Load Wallet (External -> Alviere)

* **1A (Card Load):** Plaid Token -> TabaPay AFT -> Alviere FBO Credit. User links debit card. System calculates gross charge to yield requested net balance, applying the 1.75% + $0.50 fee. TabaPay pulls funds instantly.
* **1B (Bank Load):** Plaid Token -> Dwolla RfP -> Alviere FBO Credit. User links bank. System initiates Request for Payment over RTP/FedNow. Funds settle instantly into the ledger.

### Flow 2: Peer-to-Peer Transfers (Alviere -> Alviere)

* **2A (Pre-Funded P2P):** User enters amount in keyboard. Backend verifies sufficient local wallet balance, deducts it to prevent double-spending, and creates a locked `intentId`. Keyboard injects `https://moneymsg.app/pay?id={intentId}` into RCS. Recipient taps link, opening the Companion App. Backend executes zero-cost Alviere book transfer to recipient's wallet.
* **2B (Card/Bank Funded P2P):** User enters amount in keyboard. Backend executes Flow 1 (Load) for the target send amount, charging the platform fee on the funding leg. Upon funding success, Backend verifies sufficient local wallet balance, deducts it to prevent double-spending, and creates a locked `intentId`. Keyboard injects `https://moneymsg.app/pay?id={intentId}` into RCS. Recipient taps link, opening the Companion App. Backend executes zero-cost Alviere book transfer to recipient's wallet.

### Flow 3: Cash Out (Alviere -> External)

* **3A (Card Cash Out):** Alviere FBO Debit -> TabaPay OCT. User requests withdrawal. System deducts 1.75% + $0.50 fee. TabaPay pushes net funds to the linked debit card within 30 minutes via Visa Direct/Mastercard Send.
* **3B (Bank Cash Out):** Alviere FBO Debit -> Dwolla Push. User requests withdrawal. System deducts fee and pushes net funds instantly via RTP or FedNow to the linked bank account.

## VII. Backend API Specification (Bun)

The system exposes the following REST endpoints to interface with the Kotlin Keyboard and React Native Companion App.

* `POST /api/v1/wallet/load` - Executes Flow 1A/1B using TabaPay or Dwolla.
* `POST /api/v1/wallet/cashout` - Executes Flow 3A/3B using TabaPay or Dwolla.
* `POST /api/v1/transfer/intent/wallet` - (Keyboard) Validates existing balance, holds funds, and returns the deep link intent ID for Flow 2A.
* `POST /api/v1/transfer/funded-p2p` - (Keyboard) Executes the JIT load (Flow 1), immediately holds the newly settled funds, and returns the deep link intent ID for Flow 2B.
* `POST /api/v1/transfer/claim` - (Companion App) Resolves the deep link intent ID, triggering the Alviere book transfer to the recipient for both 2A and 2B.

### Intent Expiration & Reversal Architecture (24-Hour TTL)

To enforce the 24-hour time-to-live (TTL) on unclicked RCS deep links, the database requires an explicit expiration column and an indexed state. The application layer requires an autonomous sweeping worker to identify expired holds, reverse the local ledger deduction, and update the transaction state to prevent subsequent claims.

#### 1. PostgreSQL Schema Migration

Execute this migration to append the TTL tracking and optimize the sweeping query.

```sql
-- migration_add_intent_expiration.sql

-- 1. Add expiration timestamp column
ALTER TABLE transactions
ADD COLUMN expires_at TIMESTAMPTZ;

-- 2. Retroactively set expiration for any existing pending intents
UPDATE transactions
SET expires_at = created_at + INTERVAL '24 hours'
WHERE transaction_type = 'p2p_send' AND status = 'pending';

-- 3. Create a composite index to ensure the sweeping query runs in O(log N) time
CREATE INDEX idx_transactions_expiration 
ON transactions(status, expires_at) 
WHERE status = 'pending';

```

*(Note: Ensure the `/api/v1/transfer/intent/wallet` and `/api/v1/transfer/funded-p2p` endpoints are updated to insert `expires_at: NOW() + INTERVAL '24 hours'` upon intent creation).*

#### 2. Bun Sweeping Worker (Cron)

This production-ready Bun script operates as an isolated worker process. It polls the database every 5 minutes, targeting intents that have breached the 24-hour window. It processes reversals individually to avoid locking the entire `wallets` table during bulk rollbacks.

```typescript
// intentSweeper.ts
// Environment: Bun
// Execution: bun run intentSweeper.ts (run alongside the main server process)

import postgres from "postgres";

const sql = postgres(process.env.DATABASE_URL || "postgres://user:pass@localhost:5432/moneymsg");

// Configuration
const SWEEP_INTERVAL_MS = 5 * 60 * 1000; // 5 minutes

console.log("[SWEEPER] Intent expiration worker initialized. Polling every 5 minutes.");

async function sweepExpiredIntents() {
  try {
    // 1. Identify all expired, unclaimed intents
    // Using SKIP LOCKED to prevent contention if running multiple worker instances
    const expiredIntents = await sql`
      SELECT id, wallet_id, net_amount 
      FROM transactions 
      WHERE status = 'pending' 
        AND transaction_type = 'p2p_send' 
        AND expires_at <= NOW()
      FOR UPDATE SKIP LOCKED
    `;

    if (expiredIntents.length === 0) {
      return;
    }

    console.log(`[SWEEPER] Found ${expiredIntents.length} expired intents. Processing reversals...`);

    // 2. Process each reversal in isolated transactions
    let successCount = 0;
    let failCount = 0;

    for (const intent of expiredIntents) {
      try {
        await sql.begin(async (t) => {
          // Verify it is still pending (double-check within the lock)
          const [verify] = await t`
            SELECT id FROM transactions WHERE id = ${intent.id} AND status = 'pending'
          `;
          
          if (!verify) return; // Already processed or claimed

          // Return funds to Sender's available balance
          await t`
            UPDATE wallets 
            SET available_balance = available_balance + ${intent.net_amount}, 
                last_synced_at = NOW() 
            WHERE id = ${intent.wallet_id}
          `;

          // Mark intent as reversed
          await t`
            UPDATE transactions 
            SET status = 'reversed', 
                failure_reason = 'Intent expired after 24 hours',
                completed_at = NOW()
            WHERE id = ${intent.id}
          `;
        });
        successCount++;
      } catch (err) {
        console.error(`[SWEEPER] Failed to reverse intent ${intent.id}:`, err);
        failCount++;
      }
    }

    console.log(`[SWEEPER] Sweep complete. Reversed: ${successCount}, Failed: ${failCount}`);

  } catch (error) {
    console.error("[SWEEPER] Critical database error during sweep:", error);
  }
}

// Execute immediately on startup, then trigger interval
sweepExpiredIntents();
setInterval(sweepExpiredIntents, SWEEP_INTERVAL_MS);

```

### Alviere KYC Provisioning Payload & Implementation

To comply with U.S. Money Transmitter License (MTL) and Customer Identification Program (CIP) regulations, Alviere requires a strict set of PII (Personally Identifiable Information) to run automated KYC checks before provisioning an FBO account.

The React Native Companion App must collect this data securely and transmit it to the Bun backend. The backend acts as the secure intermediary, calling Alviere's API, securely handling the SSN, and mapping the resulting member and account IDs to the local PostgreSQL database.

#### 1. React Native to Bun Payload (JSON Schema)

This is the exact JSON structure the Companion App must POST to the Bun backend.

```json
{
  "userId": "uuid-of-local-user",
  "kycData": {
    "firstName": "First",
    "lastName": "Last",
    "dateOfBirth": "YYYY-MM-DD",
    "email": "user@email.com",
    "phoneNumber": "+12345678900",
    "governmentId": {
      "type": "SSN",
      "value": "000000000" 
    },
    "address": {
      "street1": "123 Main St",
      "street2": "Apt 4B",
      "city": "CityName",
      "state": "TX",
      "postalCode": "12345",
      "countryCode": "US"
    }
  }
}

```

#### 2. Bun Backend Integration & Database Mapping

The following production-ready Bun handler executes the KYC submission, processes Alviere's synchronous approval (or pending status), and initializes the user's `wallets` record.

```typescript
// kycProvisioning.ts
// Environment: Bun
// Dependencies: postgres

import postgres from "postgres";

const sql = postgres(process.env.DATABASE_URL || "postgres://user:pass@localhost:5432/moneymsg");

interface KYCRequest {
  userId: string;
  kycData: {
    firstName: string;
    lastName: string;
    dateOfBirth: string;
    email: string;
    phoneNumber: string;
    governmentId: { type: string; value: string };
    address: {
      street1: string;
      street2?: string;
      city: string;
      state: string;
      postalCode: string;
      countryCode: string;
    };
  };
}

// ---------------------------------------------------------
// ALVIERE API MOCK
// ---------------------------------------------------------
const AlviereProvider = {
  async enrollMemberAndProvisionWallet(kycPayload: any) {
    // 1. POST https://api.alviere.com/v1/members (Submit KYC)
    // 2. POST https://api.alviere.com/v1/accounts (Provision FBO Wallet for Member)
    
    // Simulating successful synchronous KYC and wallet creation
    return {
      success: true,
      kycStatus: "approved", // 'approved', 'pending_document', or 'rejected'
      memberId: `alv_mem_${crypto.randomUUID()}`,
      accountId: `alv_acc_${crypto.randomUUID()}`,
      currency: "USD"
    };
  }
};

// ---------------------------------------------------------
// ENDPOINT: SUBMIT KYC & PROVISION WALLET
// ---------------------------------------------------------
export async function handleKYCSubmission(req: Request): Promise<Response> {
  if (req.method !== "POST") return new Response("Method not allowed", { status: 405 });

  try {
    const body = await req.json() as KYCRequest;

    // 1. Transmit PII to Alviere (Do not store SSN in local DB)
    const alviereRes = await AlviereProvider.enrollMemberAndProvisionWallet(body.kycData);

    if (!alviereRes.success || alviereRes.kycStatus === 'rejected') {
      return new Response(JSON.stringify({ error: "KYC verification failed." }), { 
        status: 400, headers: { "Content-Type": "application/json" } 
      });
    }

    // 2. Persist mapped IDs to local PostgreSQL replica
    await sql.begin(async (t) => {
      // Update User record with Alviere Member ID and Status
      await t`
        UPDATE users 
        SET alviere_member_id = ${alviereRes.memberId}, 
            kyc_status = ${alviereRes.kycStatus},
            updated_at = NOW()
        WHERE id = ${body.userId}
      `;

      // Provision local wallet mirror linked to the Alviere FBO Account ID
      await t`
        INSERT INTO wallets (user_id, alviere_account_id, available_balance, currency, status)
        VALUES (${body.userId}, ${alviereRes.accountId}, 0.0000, ${alviereRes.currency}, 'active')
        ON CONFLICT (user_id) DO NOTHING
      `;
    });

    return new Response(JSON.stringify({ 
      success: true, 
      status: alviereRes.kycStatus,
      wallet_ready: alviereRes.kycStatus === 'approved' 
    }), { headers: { "Content-Type": "application/json" } });

  } catch (error: any) {
    console.error("[KYC_ERROR]:", error.message);
    return new Response(JSON.stringify({ error: "Internal server error during KYC provisioning." }), { 
      status: 500, headers: { "Content-Type": "application/json" } 
    });
  }
}
```

### Plaid Link & Processor Token Integration

To facilitate Flow 1 (Wallet Load) and Flow 3 (Cash Out) securely without triggering PCI compliance scope for the React Native client, Plaid acts as the authentication layer.

The architecture requires a two-step handshake:

1. **React Native** requests a short-lived `link_token` from the Bun backend to initialize the UI.
2. Upon user authorization, **React Native** sends the `public_token` and `account_id` back to Bun. Bun exchanges this for a permanent `access_token` and immediately generates a `processor_token` specifically formatted for either TabaPay (Cards) or Dwolla (Banks), storing it in the `funding_sources` table.

#### 1. React Native Companion App Component

This production-ready component utilizes the `react-native-plaid-link-sdk` to manage the authentication flow natively.

```tsx
// src/components/FundingSourceLinker.tsx
// Environment: React Native
// Dependencies: react-native-plaid-link-sdk

import React, { useEffect, useState } from 'react';
import { View, Button, Text, ActivityIndicator, Alert } from 'react-native';
import { PlaidLink, LinkSuccess, LinkExit } from 'react-native-plaid-link-sdk';

interface FundingSourceLinkerProps {
  userId: string;
  sessionToken: string;
  routingTarget: 'tabapay' | 'dwolla'; // Defines the required processor token
  onSuccess: () => void;
}

export const FundingSourceLinker: React.FC<FundingSourceLinkerProps> = ({ 
  userId, 
  sessionToken, 
  routingTarget,
  onSuccess 
}) => {
  const [linkToken, setLinkToken] = useState<string | null>(null);
  const [loading, setLoading] = useState<boolean>(true);

  useEffect(() => {
    const createLinkToken = async () => {
      try {
        const response = await fetch('https://api.moneymsg.app/api/v1/plaid/create-link-token', {
          method: 'POST',
          headers: {
            'Content-Type': 'application/json',
            'Authorization': `Bearer ${sessionToken}`
          },
          body: JSON.stringify({ userId, routingTarget })
        });
        
        const data = await response.json();
        if (data.link_token) {
          setLinkToken(data.link_token);
        } else {
          Alert.alert('Error', 'Failed to initialize Plaid connection.');
        }
      } catch (error) {
        Alert.alert('Network Error', 'Unable to reach the server.');
      } finally {
        setLoading(false);
      }
    };

    createLinkToken();
  }, [userId, routingTarget, sessionToken]);

  const handlePlaidSuccess = async (success: LinkSuccess) => {
    setLoading(true);
    try {
      const response = await fetch('https://api.moneymsg.app/api/v1/plaid/exchange', {
        method: 'POST',
        headers: {
          'Content-Type': 'application/json',
          'Authorization': `Bearer ${sessionToken}`
        },
        body: JSON.stringify({
          userId,
          publicToken: success.publicToken,
          accountId: success.metadata.accounts[0].id,
          routingTarget, // 'tabapay' or 'dwolla'
          institutionName: success.metadata.institution?.name || 'Unknown Bank',
          mask: success.metadata.accounts[0].mask
        })
      });

      if (response.ok) {
        onSuccess();
      } else {
        const errData = await response.json();
        Alert.alert('Linking Failed', errData.error || 'Failed to securely store funding source.');
      }
    } catch (error) {
      Alert.alert('Network Error', 'Failed to complete linking process.');
    } finally {
      setLoading(false);
    }
  };

  const handlePlaidExit = (exit: LinkExit) => {
    console.log('User exited Plaid Link:', exit.error?.message);
  };

  if (loading) return <ActivityIndicator size="large" color="#0000ff" />;
  if (!linkToken) return <Text>Failed to load Plaid configuration.</Text>;

  return (
    <View style={{ padding: 20 }}>
      <PlaidLink
        tokenConfig={{ token: linkToken }}
        onSuccess={handlePlaidSuccess}
        onExit={handlePlaidExit}
      >
        <View style={{ backgroundColor: '#000', padding: 15, borderRadius: 8 }}>
          <Text style={{ color: '#fff', textAlign: 'center', fontWeight: 'bold' }}>
            {routingTarget === 'tabapay' ? 'Link Debit Card' : 'Link Bank Account'}
          </Text>
        </View>
      </PlaidLink>
    </View>
  );
};

```

#### 2. Bun Backend Token Exchange & Persistence

This Bun implementation handles the dual-request required by Plaid (Public Token -> Access Token -> Processor Token) and securely persists the routing mapping. It uses native `fetch` rather than the bloated Plaid Node SDK to maximize processing speed.

```typescript
// plaidIntegration.ts
// Environment: Bun
// Dependencies: postgres

import postgres from "postgres";

const sql = postgres(process.env.DATABASE_URL || "postgres://user:pass@localhost:5432/moneymsg");

// Plaid Configuration (Sandbox/Production URLs configured via ENV)
const PLAID_CLIENT_ID = process.env.PLAID_CLIENT_ID!;
const PLAID_SECRET = process.env.PLAID_SECRET!;
const PLAID_ENV_URL = process.env.PLAID_ENV_URL || "https://sandbox.plaid.com";

interface CreateLinkTokenReq {
  userId: string;
  routingTarget: 'tabapay' | 'dwolla';
}

interface ExchangeTokenReq {
  userId: string;
  publicToken: string;
  accountId: string;
  routingTarget: 'tabapay' | 'dwolla';
  institutionName: string;
  mask: string;
}

export async function handlePlaidRoutes(req: Request): Promise<Response> {
  const url = new URL(req.url);
  const path = url.pathname;

  try {
    // ---------------------------------------------------------
    // 1. CREATE LINK TOKEN (Called to initialize UI)
    // ---------------------------------------------------------
    if (req.method === "POST" && path === "/api/v1/plaid/create-link-token") {
      const body = await req.json() as CreateLinkTokenReq;

      const plaidRes = await fetch(`${PLAID_ENV_URL}/link/token/create`, {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({
          client_id: PLAID_CLIENT_ID,
          secret: PLAID_SECRET,
          client_name: "MoneyMsg",
          country_codes: ["US"],
          language: "en",
          user: { client_user_id: body.userId },
          // Dwolla requires 'auth', TabaPay requires 'auth' or 'transfer' depending on card setup
          products: ["auth"], 
        })
      });

      const data = await plaidRes.json();
      if (!plaidRes.ok) throw new Error(data.error_message || "Failed to generate link token");

      return new Response(JSON.stringify({ link_token: data.link_token }), {
        headers: { "Content-Type": "application/json" }
      });
    }

    // ---------------------------------------------------------
    // 2. EXCHANGE PUBLIC TOKEN & GENERATE PROCESSOR TOKEN
    // ---------------------------------------------------------
    if (req.method === "POST" && path === "/api/v1/plaid/exchange") {
      const body = await req.json() as ExchangeTokenReq;

      // Step A: Exchange Public Token for Access Token
      const exchangeRes = await fetch(`${PLAID_ENV_URL}/item/public_token/exchange`, {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({
          client_id: PLAID_CLIENT_ID,
          secret: PLAID_SECRET,
          public_token: body.publicToken
        })
      });

      const exchangeData = await exchangeRes.json();
      if (!exchangeRes.ok) throw new Error(exchangeData.error_message || "Token exchange failed");

      const accessToken = exchangeData.access_token;
      const itemId = exchangeData.item_id;

      // Step B: Create Processor Token (TabaPay or Dwolla)
      const processorRes = await fetch(`${PLAID_ENV_URL}/processor/token/create`, {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({
          client_id: PLAID_CLIENT_ID,
          secret: PLAID_SECRET,
          access_token: accessToken,
          account_id: body.accountId,
          processor: body.routingTarget // Exactly matches 'tabapay' or 'dwolla'
        })
      });

      const processorData = await processorRes.json();
      if (!processorRes.ok) throw new Error(processorData.error_message || "Processor token generation failed");

      const processorToken = processorData.processor_token;

      // Step C: Persist to MoneyMsg Database
      await sql`
        INSERT INTO funding_sources (
          user_id, source_type, institution_name, mask, 
          plaid_item_id, processor_token, routing_provider, is_active
        ) VALUES (
          ${body.userId}, 
          ${body.routingTarget === 'tabapay' ? 'card' : 'bank'}, 
          ${body.institutionName}, 
          ${body.mask}, 
          ${itemId}, 
          ${processorToken}, 
          ${body.routingTarget}, 
          TRUE
        )
      `;

      return new Response(JSON.stringify({ success: true }), {
        headers: { "Content-Type": "application/json" }
      });
    }

    return new Response("Not Found", { status: 404 });

  } catch (error: any) {
    console.error(`[PLAID_ERROR] ${path}:`, error.message);
    return new Response(JSON.stringify({ error: error.message }), { 
      status: 400, headers: { "Content-Type": "application/json" } 
    });
  }
}
```
